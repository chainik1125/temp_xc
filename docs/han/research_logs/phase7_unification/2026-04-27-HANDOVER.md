### CRITICAL: Three S-related blunders we made (DON'T REPEAT)

#### Blunder 1: original (T, S) validity rule was over-aggressive

Original code: `kept_windows = S - 2T + 2`, `cell_validity = (S >= 2T - 1)`.
Justified as "drop first T-1 windows because their tokens have less
preceding tail context."

**That was wrong.** Window archs encode the window IN ISOLATION at
probe time (no recurrent state). The window IS the encoder input.
No "preceding context" needed.

**Correct rule** (now in code):
```
kept_windows = effective_tail - T + 1
cell_validity = (S >= T)
```

#### Blunder 2: probe cache built at LAST_N=128 even though we now use S=32

Originally planned multi-S sweep `(128, 64, 20)` so cache was built
at LAST_N=128 with `mlc_tail` shape (N, 128, 5, d) = 17.5 GB per
task. After dropping to S=32 only, the encode still processed all
128 positions per example → only ~32 used → **~4× wasted compute**.

**FIXED** by `experiments/phase7_unification/rebuild_probe_cache_s32.py`
which slices the existing cache per-example to `(N, 32, *)`
left-aligned (real tokens right-aligned, zeros padding the start for
short sentences). New cache at `results/probe_cache_S32/`.

#### Blunder 3: WRONG simple-slice approach almost broke short sentences

Naive `cache[:, -32:, :]` drops short sentences (their real tokens are
at positions [0, L−1] for right-padded; positions [-32:] are all
padding for L<96). The user caught this.

**Correct approach** (now in code): per-example `take last min(32, n_real)
real tokens, left-pad with zeros to 32`. Records `first_real_idx` per
example so `aggregate_s` masks out windows overlapping the zero prefix.
