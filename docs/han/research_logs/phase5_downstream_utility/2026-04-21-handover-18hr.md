### TFA audit finding (BLOCKING)

Han's specific question: "Is the current TFA comparison apples-to-
apples?" Answer: **no**.

**Bug**: `experiments/phase5_downstream_utility/probing/run_probing.py:729`

```python
novel = inter["novel_codes"]   # (B, L, d_sae)
# ... this is then used as the probe input
```

**Why it's wrong**: TFA's forward (`src/architectures/_tfa_module.py:232`):

```python
x_recons = torch.matmul(z_novel + z_pred, self.D) + self.b
```

The "effective per-token representation" is `z_novel + z_pred`
(element-wise sum of sparse novel + dense pred). Current probing
uses **only** `z_novel` — strips TFA's defining mechanism
(context-predicted features via attention).

Other bench SAEs (TXCDR / MLC / Matryoshka) probe the full latent.
So TFA is probed with a **fraction** of its expressive output —
explains its rock-bottom AUC (0.6402 / 0.6346 at last_position k=5
vs the 0.78–0.80 top tier).

**Fix**: patch `run_probing.py` to encode `z_novel + z_pred`
(element-wise sum in d_sae space). Keep `scale`-multiplication on
the input; that's a TFA training-time invariant from the preload
scale.

**Diff**:

```python
# old (line ~729):
novel = inter["novel_codes"]

# new:
novel = inter["novel_codes"] + inter["pred_codes"]
```

(Keep the variable name `novel` for minimum patch surface — or
rename to `z_eff` and update the surrounding two lines.)

**Second TFA issue — capacity**: tfa_small runs at
`d_sae=4096 / seq_len=32` vs other archs at `d_sae=18432 /
seq_len=128`. `summary.md` already flags this as a caveat. For a
fair final bench, train a `tfa_big` variant.

**Training cost** (measured) for tfa_small: 155s to plateau at
4000 steps. Scaling to d_sae=18432 × seq_len=128: 4.5× (d_sae) ×
4× (seq_len) ≈ 18× per-step = ~45 min. Doable.

### Critical gotchas you must avoid

1. **`pgrep -f` combined regex broken**: `run_autoresearch.sh` used
   to have `pgrep -f "run_probing.py\|train_primary_archs"` which
   silently failed (bash escapes + pgrep BRE). Fixed at commit
   `a0339f2` to two chained `|| pgrep -f "X"` checks. If you edit
   the script, don't re-introduce this bug.

2. **`probe_val_if_missing` threshold**: in `run_autoresearch.sh`,
   `MIN_VAL_TASKS=30` — if fewer than 30 val rows exist for a base
   arch at k=5, the orchestrator re-probes the base. Don't lower
   without reason; setting to 1 re-introduces the n_paired=1
   verdict bug from commit `51659e6`.

3. **Probing is non-deterministic at fixed seed**: the top-k-by-
   class-separation tie-break is non-deterministic. Repeated probes
   of the same ckpt disagree on ~2% of rows by >0.005 AUC.
   JSONL keeps all rows; last-write-wins on duplicates for
   summariser. Don't panic if numbers shift slightly on a re-probe.

4. **Multilayer buffer is 18 GB on GPU**. Only one MLC-family
   training run at a time on A40 48GB. OOM pattern: `ml_buf` is
   kept alive between arches in `run_all`; add `del ml_buf;
   ml_buf = None` after each MLC training if you need the memory
   back.

5. **Don't skip plateau stop** unless specifically doing the 50k
   training-budget check. Plateau was hit for all A2/A3 variants
   by step 5200-7500.

6. **Test split is ONLY touched via `partB_finalize --run`** or by
   running `run_probing.py --aggregation last_position/mean_pool`
   (no `_val` suffix). Val mode (`last_position_val`) is the
   default for all autoresearch iterations. If you accidentally
   touch test while tuning, commit a note in the JSONL.

7. **cgroup memory limit 46 GB on this pod**. Page cache hugs the
   limit while probe_cache streaming runs — failcnt stays 0, not a
   concern. Python RSS stays 7–9 GB during probing. Don't be
   alarmed by `free -h` showing low "available".
