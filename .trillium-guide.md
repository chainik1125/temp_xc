# Trillium HPC Guide

Quick reference for running jobs on the Alliance (Compute Canada) Trillium GPU cluster.

## Documentation

- **Trillium Quickstart**: https://docs.alliancecan.ca/wiki/Trillium_Quickstart
- **Alliance general docs**: https://docs.alliancecan.ca/wiki/Technical_documentation
- **SLURM reference**: https://docs.alliancecan.ca/wiki/Running_jobs

## SSH Access

```bash
ssh aniketrd@trillium-gpu.scinet.utoronto.ca
```

Add to `~/.ssh/config` for convenience:

```
Host trillium
    HostName trillium-gpu.scinet.utoronto.ca
    User aniketrd
```

Then just `ssh trillium`.

## Filesystem

| Path | Purpose | Quota | Backed up? |
|------|---------|-------|------------|
| `$HOME` | Config, envs, small scripts | 50 GB | Yes |
| `$SCRATCH` | Working data, results, code | Large | No (purged after 60 days) |
| `$PROJECT` | Shared group storage | Per-group | Yes |

Your code typically lives at `$SCRATCH/<project-name>`.

## Trillium-Specific Constraints

Trillium is different from Narval/Cedar/Graham:

1. **16 CPUs, 384 GB RAM, 4x RTX 2070 per node** (one node has an RTX 5000 Ada)
2. **1 GPU = 4 CPUs**: request `--gpus-per-node=1 --cpus-per-task=4` unless your code uses multiple GPUs
3. **No `--mem` flag**: memory is fixed per node
4. **Max walltime**: 24 hours (`--time=24:00:00`)
5. **`--export=NONE` default**: environment variables from login shell are NOT inherited. Use `module load` inside your script. `$HOME` and `$SCRATCH` are set by the system.

## Environment Setup (one-time)

```bash
# On Trillium login node
module load StdEnv/2023 python/3.11 cuda/12.2 scipy-stack/2024a
python3 -m venv $HOME/envs/myenv
source $HOME/envs/myenv/bin/activate
pip install torch pennylane h5py pyyaml tqdm
```

## Submitting Jobs

### Inline command (`--wrap`)

Good for simple one-liners. Watch the character limit — Trillium truncates long `--wrap` strings.

```bash
sbatch \
  --partition=compute_full_node \
  --gpus-per-node=4 \
  --cpus-per-task=8 \
  --time=04:00:00 \
  --account=rrg-aspuru \
  --job-name=my-job \
  --output=logs/my-job_%j.out \
  --error=logs/my-job_%j.err \
  --wrap='
module load StdEnv/2023 python/3.11 cuda/12.2 scipy-stack/2024a
source "$HOME/envs/myenv/bin/activate"
cd "$SCRATCH/my-project"
export PYTHONUNBUFFERED=1
python3 train.py --config config.yaml
'
```

### Script file (recommended for anything non-trivial)

Write a bash script and submit it:

```bash
cat > run.sh << 'EOF'
#!/bin/bash
set -euo pipefail
module load StdEnv/2023 python/3.11 cuda/12.2 scipy-stack/2024a
source "$HOME/envs/myenv/bin/activate"
cd "$SCRATCH/my-project"
export PYTHONUNBUFFERED=1

python3 train.py --config config.yaml
python3 evaluate.py --checkpoint checkpoint.pt
EOF
chmod +x run.sh

sbatch \
  --partition=compute_full_node \
  --gpus-per-node=4 \
  --cpus-per-task=8 \
  --time=24:00:00 \
  --account=rrg-aspuru \
  --job-name=my-job \
  --output=logs/my-job_%j.out \
  --error=logs/my-job_%j.err \
  run.sh
```

### Job dependencies (chaining)

```bash
JOB1=$(sbatch --parsable ... job1.sh)
JOB2=$(sbatch --parsable --dependency=afterok:$JOB1 ... job2.sh)
echo "job1=$JOB1 -> job2=$JOB2"
```

`afterok` = run job2 only if job1 exits with code 0. Other options: `afterany` (run regardless), `afternotok` (run only if job1 fails).

## Monitoring Jobs

```bash
# Your running/pending jobs
squeue -u $USER

# Detailed job info
scontrol show job <JOBID>

# Completed job history (today)
sacct -u $USER --starttime=today \
  --format=JobID%12,JobName%25,State%12,ExitCode,Elapsed

# All jobs since a date
sacct -u $USER --starttime=2026-04-01 \
  --format=JobID%12,JobName%25,State%12,ExitCode,Elapsed,MaxRSS

# Cancel a job
scancel <JOBID>

# Cancel all your jobs
scancel -u $USER

# Check account allocation/usage
sshare -A rrg-aspuru -u $USER
```

## Tailing Logs

Logs are written to `--output` and `--error` paths. Tail them while the job runs:

```bash
# From Trillium login node
tail -f logs/my-job_12345.out

# Last 50 lines of stderr (errors/tracebacks)
tail -50 logs/my-job_12345.err

# Search for errors
grep -i "error\|traceback\|failed" logs/my-job_12345.err
```

## Copying Results Back Locally

```bash
# Single file
scp trillium:'$SCRATCH/my-project/results/output.json' ./results/

# Directory (recursive)
scp -r trillium:'$SCRATCH/my-project/results/' ./results/

# Rsync (incremental, faster for repeated fetches)
rsync -avz trillium:'$SCRATCH/my-project/results/' ./results/

# Note: $SCRATCH won't expand in single quotes over scp.
# Either use the full path or wrap in double quotes:
scp trillium:"/scratch/aniketrd/my-project/results/file.json" .
```

### Fetch script pattern

```bash
#!/bin/bash
# fetch_results.sh — run locally
REMOTE="aniketrd@trillium-gpu.scinet.utoronto.ca"
REMOTE_DIR="\$SCRATCH/my-project"
LOCAL_DIR="$(cd "$(dirname "$0")/.." && pwd)"

rsync -avz "${REMOTE}:${REMOTE_DIR}/results/" "${LOCAL_DIR}/results/"
rsync -avz "${REMOTE}:${REMOTE_DIR}/logs/" "${LOCAL_DIR}/logs/" 2>/dev/null || true
```

## Pushing Code

```bash
# On Trillium — pull from git
cd $SCRATCH/my-project
git pull

# Or rsync from local (if not using git)
rsync -avz --exclude='results' --exclude='*.h5' \
  ./my-project/ trillium:'$SCRATCH/my-project/'
```

## Common Gotchas

1. **`--wrap` length limit**: Trillium silently truncates long `--wrap` strings. If your job fails with cryptic errors, write a script file instead.

2. **Logs not streaming**: Add `export PYTHONUNBUFFERED=1` at the top of your script. Without it, Python buffers stdout and logs only appear after the job finishes.

3. **`--mem` not allowed**: Don't use `--mem`, `--mem-per-cpu`, etc. Memory is fixed per node.

4. **Single GPU not possible**: Every job gets 4 GPUs. Your code only needs to use one — the others sit idle. Set `CUDA_VISIBLE_DEVICES=0` if your code tries to use all GPUs and you don't want that.

5. **Module load order matters**: Always load `StdEnv/2023` first, then `python/3.11`, then `cuda/12.2`.

6. **Scratch purge**: Files on `$SCRATCH` untouched for 60 days get deleted. Copy important results to `$PROJECT` or locally.

7. **`set -euo pipefail`**: Always use this in scripts. Without it, a failed command (e.g., `cd` to wrong dir) won't stop the script, and you'll get confusing errors downstream.

## Template: Full Pipeline Script

```bash
#!/bin/bash
# submit_pipeline.sh — run from Trillium login node
set -euo pipefail

PARTITION="compute_full_node"
ACCOUNT="rrg-aspuru"

# Job 1: preprocessing
JOB1=$(sbatch --parsable \
  --partition=${PARTITION} --gpus-per-node=4 --cpus-per-task=8 \
  --time=04:00:00 --account=${ACCOUNT} \
  --job-name=preprocess --output=logs/preprocess_%j.out --error=logs/preprocess_%j.err \
  scripts/preprocess.sh)
echo "Preprocess: $JOB1"

# Job 2: training (after preprocessing)
JOB2=$(sbatch --parsable \
  --dependency=afterok:${JOB1} \
  --partition=${PARTITION} --gpus-per-node=4 --cpus-per-task=4 \
  --time=24:00:00 --account=${ACCOUNT} \
  --job-name=train --output=logs/train_%j.out --error=logs/train_%j.err \
  scripts/train.sh)
echo "Train: $JOB2 (after $JOB1)"

# Job 3: evaluation (after training)
JOB3=$(sbatch --parsable \
  --dependency=afterok:${JOB2} \
  --partition=${PARTITION} --gpus-per-node=4 --cpus-per-task=4 \
  --time=04:00:00 --account=${ACCOUNT} \
  --job-name=eval --output=logs/eval_%j.out --error=logs/eval_%j.err \
  scripts/eval.sh)
echo "Eval: $JOB3 (after $JOB2)"

echo "Monitor: squeue -u \$USER"
```
